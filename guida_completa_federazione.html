Guida Completa: Federazione OpenStack Keystone ↔ Keycloak

Questo documento raccoglie le specifiche tecniche dell'infrastruttura, i parametri di configurazione e le soluzioni applicate per risolvere le criticità riscontrate (Errore 500 e permessi mancanti).

1. Panoramica Infrastruttura

  • Protocollo: OIDC (OpenID Connect)
  • Flusso: Keycloak (IdP) → Apache mod_auth_openidc → Keystone (SP)
  • Stato Attuale: OPERATIVO

2. Parametri Ambientali (Environment Constants)

Componente Parametro Valore Note
Apache OIDC Claim Prefix OIDC- Prefisso standard per i claim
Keycloak Realm openstack
Keycloak Client keystone
Keystone Federated Domain federated_domain ID: e289...
Keystone Local Group federated_users ID: 0d98...

3. Configurazione Keycloak (IdP)

È stato configurato un Mapper nel client keystone per esporre i gruppi nel token JWT.

  • Mapper Name: group-mapper
  • Mapper Type: Group Membership
  • Token Claim Name: groups
  • Full group path: OFF (ritorna solo il nome del gruppo, es. os-admins)
  • Add to ID/Access Token: ON

4. Troubleshooting e Soluzioni Applicate

Storico Problemi:
  1. Errore 401: Risolto inizialmente tramite mapping.
  2. Errore 500 (Login Failed): Keystone falliva nella creazione dell'utente effimero.
  3. Permessi Insufficienti: L'utente accedeva ma non vedeva le risorse IoT.

A. Risoluzione Errore 500 (Mapping Definitivo)

Causa: L'uso di ID statici nel mapping JSON (tentativo precedente) e l'attributo OIDC-preferred_username causavano conflitti o valori nulli.

Soluzione: Siamo passati a un mapping basato sui Nomi (più leggibile e portabile) e sull'attributo OIDC-sub (Subject ID), che è l'identificativo univoco garantito da Keycloak.

Configurazione Mapping Attiva (rules.json / Keystone):

[
    {
        "local": [
            {
                "user": {
                    "name": "{0}",
                    "domain": { "name": "federated_domain" }
                },
                "group": {
                    "name": "federated_users",
                    "domain": { "name": "federated_domain" }
                }
            }
        ],
        "remote": [
            { "type": "OIDC-sub" }
        ]
    }
]

Comando per applicare il mapping:

openstack mapping set --rules rules.json keycloak_mapping

B. Ripristino Permessi Dashboard

Problema: Il solo ruolo admin non era sufficiente per vedere le board e le flotte Iotronic.

Soluzione: Sono stati assegnati manualmente i ruoli specifici di Iotronic al gruppo federated_users sul progetto admin.

Comandi Eseguiti (Container Keystone):

# 1. Assegnazione ruoli Iotronic
openstack role add --group federated_users --group-domain federated_domain --project admin --project-domain Default admin_iot_project
openstack role add --group federated_users --group-domain federated_domain --project admin --project-domain Default manager_iot_project
openstack role add --group federated_users --group-domain federated_domain --project admin --project-domain Default user_iot

# 2. Verifica Assegnazioni
openstack role assignment list --group federated_users --group-domain federated_domain --names

C. Riepilogo Ruoli Attivi

  • admin: Amministrazione generale OpenStack.
  • admin_iot_project: Privilegi amministrativi completi per IoT.
  • manager_iot_project: Gestione operativa risorse IoT.
  • user_iot: Accesso base utente IoT.
Stato Finale: Login funzionante via Keycloak e accesso completo alla Dashboard Iotronic.
